# /新需求 全自动流水线

用户输入：$ARGUMENTS

**核心原则**：用户只在这一刻说一次需求，**之后全自动跑到底，只在终点汇报**。
**不要在中途列任务清单等用户确认**。**不要在每条 PASS 后停下问下一条**。
唯一允许停下的 4 个口子见 CLAUDE.md §7。

按以下 5 步严格执行：

---

## Step 1：理解需求 + 红线预检

1. 读 `docs/产品文档.md` 全文
2. 读 4 个任务文件总览段
3. 把 `$ARGUMENTS` 解析成结构化需求：
   - 影响哪些模块（user / content / channel_pack / admin / 跨模块）
   - 涉及哪些角色（后端 / 管理后台 / 安卓 / 测试）
   - 是新增 / 修改 / 删除 / 改字段 / 改流程
4. 对照 `docs/架构与红线.md` 第 5 章「红线」+ 第 6 章「不做清单」+ 第 9 章「红蓝对抗触发条件」
   - **不触线** → 进入 Step 2
   - **触线（必停口子①）** → **立即停下**：
     - 调 `redblue` subagent 给合稿
     - 把合稿（方案 / 风险 / 缓解 / 是否破例 / 推荐意见）原样输出给用户
     - **结束本次流水线**，等用户拍板后由用户决定是否重启

5. 简短汇报（一句话）：「需求理解：X，影响 Y 模块。开始全自动流水线。」

## Step 2：改产品文档

1. 在 `docs/产品文档.md` 对应章节追加 / 修改 / 删除（用 Edit 工具，**不要 Write 整个文件**）
2. 改第 5 章「当前实现状态」表格，把新需求标 ❌ 0/N
3. 必要时改第 6 章「后续规划建议」
4. 不新建顶层章节，除非确实是新模块 → 按现有 4.X 子节风格扩展

## Step 3：planner 拆任务

```
Agent({
  description: "拆解新需求到 4 角色任务文件",
  subagent_type: "planner",
  prompt: "<结构化需求 + 改后的产品文档章节链接>"
})
```

planner 返回任务列表。检查：
- **0 条任务（必停口子②）** → 停下汇报「planner 拆出 0 条，需求点 X / Y 不清晰，请补充」结束
- **任务间互相冲突 / 跨多角色但缺关键依赖（必停口子②）** → 同上
- **任务全部 OK** → 进入 Step 4（**不列清单等用户确认**，直接开干）

## Step 4：自动 for-loop 实施所有任务

按依赖顺序排好任务列表（无依赖优先）。然后：

```python
for task in tasks_in_dependency_order:
    attempt = 1
    last_dev_agent_name = None

    while attempt <= 3:
        # 调 developer
        if attempt == 1:
            dev_result = Agent(
                description=f"实施任务 {task.id}",
                subagent_type="developer",
                name=f"dev-{task.id}",  # 命名以便 SendMessage resume
                prompt=f"<任务规格 + 验收 + 专属测试规格>"
            )
            last_dev_agent_name = f"dev-{task.id}"
        else:
            # FAIL 重试时复用上下文
            dev_result = SendMessage(
                to=last_dev_agent_name,
                message=f"tester 报告 FAIL（第 {attempt-1} 轮）：\n{tester_report}\n请修复后回复 done。"
            )

        # developer 中途如果汇报"任务规格歧义"（必停口子③）→ 中断流水线
        if dev_result.contains("规格歧义"):
            return STOP_AND_REPORT(reason="dev 发现歧义", task=task, dev_msg=dev_result)

        # 调 tester
        tester_result = Agent(
            description=f"校验任务 {task.id}",
            subagent_type="tester",
            prompt=f"<任务 ID + dev 改动文件 + 专属测试路径>"
        )

        if tester_result.PASS:
            mark_task_done_in_file(task.id)         # [ ] → [✓]
            update_progress_total()                 # 更新角色任务文件「进度总览」数字
            git_add_and_commit(f"done: {task.id} {task.short_desc}")
            break  # 进下一条
        else:
            tester_report = tester_result.report
            attempt += 1

    if attempt > 3:
        # FAIL 3 轮（口子④：不停，标 [~] 累积进终报告）
        mark_task_skipped_in_file(task.id, status="[~]", note="3 轮 FAIL，见终报告")
        skipped_tasks.append((task, last_3_reports, dev_root_cause_guess))
        # 不 break、不 stop —— 进下一条
```

实施细节：
- 每条 PASS 后 commit 一次（**不要批量 commit**）
- commit message：`done: <ID> <一句话描述>` 沿用项目历史风格
- API 改动后 dev 会自己跑 `scripts/export_api_docs.py`（developer agent 内置）
- import-linter / pytest / pnpm build 全在 tester 里跑（tester agent 内置）

## 必停口子总览（违反就破坏全自动诺言）

| ID | 触发 | 处理 |
|---|---|---|
| ① | Step 1 触红线 | 调 redblue → 输出合稿 → 结束流水线等用户拍板 |
| ② | Step 3 planner 拆解失败 | 列出歧义点 → 结束流水线等用户补需求 |
| ③ | Step 4 dev 发现规格歧义 | 中断 for-loop → 列出问题 → 等用户改产品文档 |
| ④ | Step 4 同任务 FAIL 3 轮 | **不停**，标 `[~]` 累积进终报告，进下一条 |

口子 ④ 是关键设计：3 轮死循环时**绝不打断**整个流水线，把失败累积到终报告，符合"我只看终验"诉求。
