# /新需求 流水线

用户输入：$ARGUMENTS

按以下 6 步严格执行，**每一步都要在主对话给用户简短汇报**，不准默默推进。

---

## Step 1：理解 + 现状对照

1. 读 `docs/产品文档.md` 全文
2. 读 4 个任务文件总览段
3. 把 `$ARGUMENTS` 解析成结构化需求：
   - 影响哪些模块（user / content / channel_pack / admin / 跨模块）
   - 涉及哪些角色（后端 / 管理后台 / 安卓 / 测试）
   - 是新增 / 修改 / 删除 / 改字段 / 改流程
4. 汇报：「我理解的需求是 X，影响 Y 模块 Z 角色。开始走流程。」

## Step 2：红线预检

对照 `docs/架构与红线.md` 第 5 章 + 第 6 章 + 第 9 章。

- **不触线** → 进入 Step 3
- **触线** → **立即停下**，调 `redblue` subagent 开红蓝对抗，把合稿（方案 / 风险 / 缓解 / 是否破例 / 推荐意见）给用户拍板。**不准擅自继续**。

## Step 3：改产品文档

1. 在 `docs/产品文档.md` 对应章节追加 / 修改 / 删除（**不要新建顶层章节**除非确实是新模块）
2. 如果是新模块，按现有结构（4.X 子节）扩展
3. 改第 5 章「当前实现状态」表格，把新需求标 ❌ 0/N
4. 必要时改第 6 章「后续规划建议」
5. 用 Edit 工具，**不要 Write 整个文件**

## Step 4：调 planner subagent 拆任务

调用方式：
```
Agent({
  description: "拆解新需求到 4 角色任务文件",
  subagent_type: "planner",
  prompt: "<把 Step 1 解析的结构化需求 + Step 3 改后的产品文档章节链接传过去>"
})
```

planner 会返回每个角色文件要追加的任务条目，每条带：
- 任务 ID（沿用 P 阶段编号或新增 N.X）
- 一句话描述
- 验收标准
- **专属测试用例规格**（Q3 强约束）
- 依赖关系
- 红线提示（如有）

## Step 5：列任务清单 + 等用户确认

1. 把 planner 输出汇总成清单贴给用户
2. **明确问**：「本次想做哪条 / 哪几条？默认只做第一条。」
3. **停下来等用户回复**。不准擅自往下走。

## Step 6：实施第一条（developer + tester 循环）

收到用户确认要做哪条后：

1. 把要做的任务写进对应任务文件（用 Edit）
2. 调 `developer` subagent 实施：
```
Agent({
  description: "实施任务 <ID> <描述>",
  subagent_type: "developer",
  name: "dev-<ID>",     // 命名以便 SendMessage 复用
  prompt: "<任务规格 + 验收标准 + 专属测试用例规格>"
})
```
3. developer 完成后调 `tester`：
```
Agent({
  description: "校验任务 <ID>",
  subagent_type: "tester",
  prompt: "<任务 ID + developer 改了哪些文件 + 专属测试文件路径>"
})
```
4. **PASS** → 任务文件 `[ ]` → `[✓]`，更新进度总览，commit `done: <ID> ...`，汇报用户「这条完成了，下一条是 X，要继续吗？」**停下**
5. **FAIL** → SendMessage 回 dev-<ID> 保留上下文修：
```
SendMessage({
  to: "dev-<ID>",
  message: "tester 报告 FAIL：<报告内容>。请修复后回复 done。"
})
```
最多 3 轮重测，3 轮还 FAIL → 停下汇报「3 轮没修好，原因 X，要不要换思路 / 跳过 / 我介入？」

---

## 异常分支

| 情况 | 处理 |
|---|---|
| 用户在 Step 5 说"全做" | 还是只做第一条，但在汇报时强调「按约定单任务做完即停，剩下我会按顺序逐条来。要继续敲 ok / 直接说"继续"」 |
| 用户在 Step 1 说"不用走流程，直接做" | 提醒约束：「项目规则要求先改产品文档。如果坚持跳过，请明确说"跳过流程"，我会记录在本次会话日志里」 |
| Step 4 planner 输出与红线冲突 | 退回 Step 2 重判，调 redblue |
| Step 6 developer 写代码时发现需求歧义 | developer 中断 → coordinator 回去问用户 → 改产品文档 → 重启 |
